## 开发工作流

执行工作流时，严格按照流程和顺序，执行到某一步时，必须**显示地**声明当前执行到是第几步！

### 第 1-5 步：计划、编码、验证

1. **理解**：先明确任务与代码库，按需搜集信息；
2. **计划**：制定详细计划，列出需修改文件；如果有todolist工具，请列出todos。todos 包含 workflow 3-5 所有内容
   向用户输出文件路径时，用相对路径（更简洁）
3. **Coding**：先获取相关代码细节，再谨慎编辑文件，代码风格保持和用户一致。如需安装新依赖，使用 npm 包管理器安装；
4. **验证**：完成后检查是否需后续修改，建议测试验证；
5. **评估改动是否需要 CodeReview**：非常小的改动，如简单的文案、配置、ui改动，不需要codereview。否则请务必执行第六步 codereview 行为。

### 第 6 步：按需进行CodeReview

在这一步中，请你通过 read_file 工具读取 `.catpaw/rules/codejoy/code-style.md` 审查**刚刚修改的代码**，优化代码，确保最终交付高质量代码。

注意事项：

- 审查范围为**刚刚修改的代码**。不要引入无关的改动。
- 因为代码质量问题导致需要重新修改时，请探索仓库已有的实现方式，评估合理新的修改方案是否符合仓库场景。

**常见代码问题检查清单：**

- [ ] 函数参数入参过多？应用 key-value 对象字面量传递多参数
- [ ] 新代码有重复、冗余？提取为函数或常量
- [ ] 代码健壮性？错误处理、边界情况、类型安全
- [ ] 魔法数字？提取为常量
- [ ] 硬编码的映射或配置？提取为常量或配置文件
- [ ] 代码注释是否有价值？移除冗余注释,保留有价值的注释

**CodeReview 完成后的输出：**

- 更新任务清单，标记 CodeReview 完成
- 记录发现的问题和优化方案
- 确保 TypeScript 类型检查通过

### 当用户指出代码问题时

1. 检查 `.catpaw/rules/codejoy/code-style.md` 代码规范是否明确，完善并记录代码规范，后续参考
2. 立即修复问题
3. 重新进行 CodeReview 确保修复正确

## 可参考文件

当需要更详细的指导时，可以查阅以下文件。使用编辑器的 `read_file` 工具（而不是 `fetch_rule` 或其他方法）来读取这些文件：

- `.catpaw/rules/codejoy/code-style.md`: 编写或修改代码时，检查代码是否遵循规范。包括：组件/函数设计原则、TypeScript 类型编程规范、代码健壮性要求、常量管理、代码简洁性、注释规范、样式管理（SCSS 模块化组织规范、BEM 命名方法、选择器嵌套原则等）。在代码审查或优化时必读。
